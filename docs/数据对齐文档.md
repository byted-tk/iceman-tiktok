    "open_id": "owner_user_123",      // 用户唯一标识（主键）
    "nickName": "冬日暖阳",            // 用户昵称
    "avatarUrl": "https://...",        // 头像 URL  ⚠️ 注意驼峰命名
    "gender": 1,                       // 0=未知 1=男 2=女
    "country": "中国",
    "province": "北京",
    "city": "北京",
    "language": "zh_CN",
    "role": "host",                    // 角色标记（仅 demo 用）: host | visitor

  }
]

**⚠️ 已知不一致**：旧版文件中用 `avatar`（无 Url 后缀），现已统一为 `avatarUrl`。  
**⚠️ demo 新增字段**：`role`, `interests`, `bio`, `iceman_nickname` 是 demo 扩展字段，开放平台原始接口没有。

## 4. 访客会话 & 消息数据

这是**最核心**的数据，存在两套文件格式（LLM 模块遗留格式 + 后端格式），需要对齐。

### 4.1 LLM 模块遗留格式

**文件路径**：`dialog_memory/filtered_conversation_{timestamp}.json`  
**写入者**：`MemoryManager.store_filtered_conversation()`

```json
{
  "conversation": [
    { "role": "user",      "content": "访客说的话" },
    { "role": "assistant", "content": "小冰回复" }
  ],
  "reason": "过滤原因",           // MemoryManager.filter_conversation() 返回的 reason
  "visitor_input": "visitor_user_456",  // ⚠️ 字段名是 visitor_input，实际存的是 visitor open_id
  "timestamp": 1743764400,       // Unix 时间戳（秒）
  "should_show_to_host": true    // MemoryManager.filter_conversation() 返回的 bool
}
```

**⚠️ 遗留问题**：
- `visitor_input` 字段名有歧义（实际存的是 open_id，不是消息内容）
- `conversation` 中每条消息没有 `timestamp`
- 没有 `session_id`，无法和后端会话关联

### 4.2 后端格式（规范版 / 唯一权威格式）

**文件路径**：`dialog_memory/session_{visitor_id}_{date}.json`  
**写入者**：后端 HTTP 服务  
**说明**：MVP 阶段此文件是唯一权威数据源；`filtered_conversation_*` 是遗留兼容格式（见 4.1）。

```json
{
  "session_id": "sess_visitor456_20260404",
  "visitor_id": "visitor_user_456",        // ← 对应 users.open_id
  "iceman_id": "iceman_owner123",
  "owner_id": "owner_user_123",
  "date": "2026-04-04",
  "status": "ai_chatting",                 // 由此字段派生 should_show_to_host（不再单独存储）
  "is_folded": false,
  "filter_reason": "",                     // ← 对应遗留格式的 reason
  "takeover_at": null,
  "messages": [
    {
      "message_id": "msg_001",
      "sender_type": "Visitor",            // Visitor | IceMan | Host | System
      "sender_id": "visitor_user_456",
      "content": "访客说的话",
      "content_type": "text",
      "timestamp": 1743764400             // ← 每条消息有独立时间戳（秒）
    },
    {
      "message_id": "msg_002",
      "sender_type": "IceMan",
      "sender_id": "iceman_owner123",
      "content": "小冰回复",
      "content_type": "text",
      "timestamp": 1743764420
    }
  ],
  "visitor_interest_tags": ["滑雪", "阿勒泰"],
  "created_at": 1743764400,
  "updated_at": 1743764420
}
```

> **`should_show_to_host` 已废弃**：不再作为单独字段存储，直接由 `status` 派生：`status != "filtered_blocked"` → 对主人可见。

### 4.3 遗留格式 → 后端格式迁移映射

| 遗留格式字段 | 后端格式字段 | 说明 |
| :--- | :--- | :--- |
| `visitor_input` | `visitor_id` | 字段语义相同，名称修正 |
| `timestamp`（顶层）| `created_at` | 会话创建时间 |
| `should_show_to_host: true` | `status: ai_chatting`（派生） | 展示状态，不再单独存 |
| `should_show_to_host: false` | `status: filtered_blocked`（派生） | 屏蔽状态 |
| `reason` | `filter_reason` | 过滤原因 |
| `conversation[].role: "user"` | `messages[].sender_type: "Visitor"` | 值语义相同，字段名统一 |
| `conversation[].role: "assistant"` | `messages[].sender_type: "IceMan"` | 同上 |
| （无）| `messages[].message_id` | 新增，后端生成唯一 ID |
| （无）| `messages[].sender_id` | 新增，Visitor/IceMan/Host 各自 open_id |

### 4.4 会话状态 × LLM 模块调用时机（⚠️ 在线/离线分离）

> **核心区分**：  
> - `classify_intent()` = **在线**，每条消息进来时同步调用 → 直接决定本条消息如何回复 + 更新 `session.status`  
> - `filter_conversation()` = **离线**，会话结束后（或定时）异步调用 → 仅用于"是否展示给主人"判断，**不影响在线回复链路**

#### 在线链路（每条访客消息）

| `classify_intent()` 返回 | 在线处理方式 | `session.status` 更新 | `filter_reason` |
| :--- | :--- | :--- | :--- |
| `BENIGN_INTERACTION` / `GENERAL_INQUIRY` | `_generate_response()` → 正常 AI 回复 | `ai_chatting` | 不写入 |
| `INAPPROPRIATE_REQUEST` | `_generate_inappropriate_response()` → 兜底回复 | `filtered_folded` | `"INAPPROPRIATE_REQUEST"` |
| `PRIVACY_SENSITIVE` | `_generate_privacy_protected_response()` → 隐私保护回复 | `filtered_folded` | `"PRIVACY_SENSITIVE"` |
| —（主人手动）| 主人点击接管按钮 | `host_takeover`（不可逆） | 不写入 |

#### 离线链路（会话结束后异步）

| `filter_conversation()` 返回 | 含义 | 对主人展示 |
| :--- | :--- | :--- |
| `(True, reason)` | 整段对话质量正常 | ✅ 正常展示 |
| `(False, reason)` + 原因为明确骚扰/广告 | 整段对话应被完全屏蔽 | ❌ `filtered_blocked`，主人不可见 |

> **注意**：`filter_conversation()` 结果**不**覆盖在线已写入的 `session.status`，只写入 `filter_reason`。若判定为 blocked，后端将 `status` 升级为 `filtered_blocked`。

### 4.5 〔二期迁移参考〕MySQL：`visitor_sessions` + `messages` 表

> **MVP 阶段不使用**，以下仅供二期数据库迁移参考。

## 6. Embedding 缓存数据

**文件路径**：`dataset/embedding_cache/emb_{hash}.json`  
**读写者**：`MemoryManager.get_embedding(text)`

```json
[0.123, -0.456, 0.789, ...]   // 纯向量数组，无元数据
```

**hash 计算**：`hash(text) % 1000000`（Python 内置 hash，**不稳定**，跨进程可能不同）

**⚠️ 注意**：Python 的 `hash()` 在不同进程/重启后结果不同（Python 3.3+ 默认开启 hash randomization）。生产环境需改为确定性 hash（如 `hashlib.md5(text.encode()).hexdigest()[:8]`）。Demo 阶段可接受。

**MongoDB 对应**：`embedding_cache` 集合，添加 `text_snippet` 字段便于 debug。

```json
{
  "summary_id": "sum_20260404_owner123",
  "owner_user_id": "owner_user_123",

## 9. 潜在连接对象数据

**文件路径**：`dialog_memory/connection_{visitor_id}_{timestamp}.json`  
**写入者**：`MemoryManager.mark_potential_connection(visitor_id, summary)`

}
```

**触发条件**：`main.py` 中的 `_check_potential_connection()` 调用 LLM 判断对话是否为优质连接，返回非空字符串则触发写入。

**与 session 数据的关联**：目前只有 `visitor_id`，没有 `session_id`。后端 API 需通过 `visitor_id` + `date` 反查对应 session。

### 13.2 PM Q8 实现指引：主人历史对话纳入 Memory

**目标**：`UserDialogueManager._generate_response()` 的 system prompt 中追加最近 N 条主人与小冰人的对话，让 AI 更了解主人近期状态。

**数据来源**（均为现有数据，无需后端新增接口）：

```python
import os, json, glob

def get_host_recent_dialogue(memory_dir: str, owner_id: str, n: int = 10):
    """从 session 文件中提取主人（Host）发言记录"""
    host_messages = []
    pattern = os.path.join(memory_dir, "session_*.json")
    for fpath in sorted(glob.glob(pattern), reverse=True):
        with open(fpath) as f:
            s = json.load(f)
        if s.get("owner_id") != owner_id:
            continue
        # 取主人（Host）发的消息
        for msg in s.get("messages", []):
            if msg.get("sender_type") == "Host":
                host_messages.append(msg["content"])
        if len(host_messages) >= n:
            break
    return host_messages[-n:]
```

追加到 `_generate_response()` 的 system_prompt 变量中即可，格式参考 §6.1 的 prompt 模板。

### 13.3 对话状态与 classify_intent 对应关系（后端视角）

后端只调用 `classify_intent()` 一次（不调 `process_user_input()`），根据返回值直接调对应方法：

```python
intent = mgr.classify_intent(content)
if intent == "INAPPROPRIATE_REQUEST":
    reply = mgr._generate_inappropriate_response(content)
    status = "filtered_folded"
elif intent == "PRIVACY_SENSITIVE":
    reply = mgr._generate_privacy_protected_response(content)
    status = "filtered_folded"
else:  # BENIGN_INTERACTION | GENERAL_INQUIRY
    reply = mgr._generate_response(content)
    status = "ai_chatting"
```

**不要**调 `process_user_input()`——它内部会再次调 `classify_intent()`，造成双重 LLM 调用。
